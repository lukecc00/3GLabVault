# 双端鉴权与权限隔离说明

## 1. 当前目标

当前阶段先完成最小可运行的登录态、路由守卫和后端权限隔离，确保：

- 用户端与管理员端都需要登录后访问
- 管理员端只允许管理员角色进入
- 后端管理接口不能被匿名直接调用
- 知识页面的作者与编辑者由当前登录用户自动推导

## 2. 当前实现方式

由于当前数据库模型中还没有密码字段，且 Keycloak 还未正式接入业务代码，因此本阶段采用：

- 前端：本地开发登录页 `/login`
- 后端：`/api/auth/login`、`/api/auth/me`、`/api/auth/logout`
- 会话：前端保存 Bearer Token，本地存储于浏览器 `localStorage`
- Token：后端基于 `AUTH_TOKEN_SECRET` 签发开发期访问令牌

说明：

- 这是开发阶段的过渡方案
- 后续接入 Keycloak 时，前端登录页和后端 Token 校验逻辑将替换为统一 OIDC 登录
- 业务侧的“当前用户 / 角色 / 群组 / 管理员端守卫”结构可继续复用

## 3. 路由访问规则

- `/`：公共首页
- `/login`：登录页
- `/portal`：用户端，需要登录
- `/portal/knowledge/*`：知识库用户侧页面，需要登录
- `/admin/*`：管理员端，需要登录且需要管理员角色
- `/knowledge/*`：兼容旧路径，重定向到 `/portal/knowledge/*`

## 4. 后端权限规则

### 4.1 公开接口

- `POST /api/auth/login`
- `POST /api/auth/logout`
- `POST /api/users/register`

### 4.2 登录后可访问

- `GET /api/auth/me`
- `GET /api/knowledge/spaces`
- `GET /api/knowledge/spaces/:id`
- `GET /api/knowledge/pages`
- `GET /api/knowledge/pages/:id`
- `POST /api/knowledge/pages`
- `PATCH /api/knowledge/pages/:id`

### 4.3 仅管理员可访问

- `GET /api/users`
- `GET /api/users/:id`
- `PATCH /api/users/:id/review`
- `PATCH /api/users/:id/roles`
- `PATCH /api/users/:id/groups`
- `GET /api/roles`
- `POST /api/roles`
- `GET /api/groups`
- `POST /api/groups`
- `POST /api/groups/:id/members`
- `GET /api/organizations/summary`
- `POST /api/knowledge/spaces`

## 5. 知识库权限规则

- `PUBLIC` 空间：所有已登录成员可见
- `GROUP_RESTRICTED` 空间：仅所属群组成员和管理员可见
- `PRIVATE` 空间：当前阶段仅管理员可见

页面写入规则：

- 新建页面时，`authorId` 和 `editorId` 由当前登录用户自动写入
- 编辑页面时，`editorId` 自动更新为当前登录用户
- 普通成员只能编辑自己创建或自己最后编辑过的页面
- 管理员可以编辑所有页面

## 6. 管理员角色

当前管理员端允许以下角色进入：

- `SUPER_ADMIN`
- `DIRECTION_ADMIN`
- `GRADE_ADMIN`

## 7. 下一步建议

下一阶段建议继续完成以下内容：

- 用 Keycloak 替换当前开发登录方案
- 增加注册审核完成后的账号绑定流程
- 为知识空间和页面补更细粒度的权限模型
- 把用户端“我的文档 / 我的通知 / 邮件协同”接入当前登录态
