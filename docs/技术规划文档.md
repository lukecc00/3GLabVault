# 实验室知识库 + 邮箱系统技术规划文档

## 1. 文档目标

本文档用于规划实验室内部知识库 + 邮箱系统的整体技术方案，目标是在尽量减少重复造轮子的前提下，构建一套适合 100 人以内实验室长期使用的内部平台。

系统需满足以下核心需求：

- 支持按方向建设知识库，如 Android、iOS、Web、后端、鸿蒙。
- 知识库内容以 Markdown 为核心格式，同时支持快捷输入和扩展语法。
- 支持用户注册、审核、分组、登录和统一身份管理。
- 支持知识库空间、目录、页面级别的权限控制。
- 支持实验室邮箱系统、群组邮箱、按方向和年级进行邮件抄送。
- 支持 Docker 化部署，降低后续部署与维护成本。

## 2. 项目定位与建设原则

### 2.1 项目定位

本系统定位为实验室内部统一协作平台，包含两个主要能力中心：

- 知识沉淀中心：用于技术文档、项目经验、入门指南、日报周报模板等内容的长期沉淀。
- 邮件协同中心：用于成员邮箱、群组通信、日报周报抄送、组织通知等场景。

### 2.2 建设原则

- 稳定优先：优先采用成熟、社区活跃、长期维护良好的开源组件。
- 少造轮子：认证、邮箱、对象存储、反向代理等基础能力尽量使用成熟方案。
- 适度自研：仅对实验室特有的业务能力进行自研，如知识库空间模型、权限组合、群组同步、Markdown 扩展语法等。
- 便于部署：所有核心组件均支持 Docker 化部署，优先采用 Docker Compose 起步。
- 便于维护：前后端和基础设施边界清晰，方便后续实验室成员接手。

## 3. 技术栈总览

### 3.1 前端技术栈

- 前端框架：Next.js
- 开发语言：TypeScript
- 样式方案：Tailwind CSS
- 组件库：shadcn/ui

选型理由：

- Next.js 适合构建门户、后台管理系统、知识库浏览页和个人中心。
- TypeScript 与后端统一语言，减少跨端协作成本。
- Tailwind CSS 与 shadcn/ui 可以快速搭建风格统一、可维护的管理界面。

### 3.2 后端技术栈

- 后端框架：NestJS
- 开发语言：TypeScript
- ORM：Prisma

选型理由：

- NestJS 模块化能力强，适合账号、权限、知识库、邮件同步、审核流等中后台业务。
- TypeScript 与前端共享语言体系，降低 DTO、接口定义和协作成本。
- Prisma 在数据建模、迁移、类型安全和开发效率方面表现优秀，适合中小团队快速迭代。

### 3.3 数据与基础设施

- 主数据库：PostgreSQL
- 缓存与异步任务：Redis + BullMQ
- 附件存储：MinIO
- 认证中心：Keycloak
- 权限引擎：Casbin
- 知识库编辑器：TipTap + Markdown 转换链
- 邮箱系统：Mailcow
- 反向代理与 HTTPS：Traefik 或 Nginx Proxy Manager
- 部署方式：Docker Compose

## 4. 总体架构设计

### 4.1 架构目标

整体架构采用“自研业务系统 + 开源基础设施组件”模式：

- 自研部分：实验室门户前端、业务后端、知识库模型、权限策略、群组同步逻辑、消息通知逻辑。
- 复用部分：统一认证、对象存储、邮件服务器、缓存队列、数据库、反向代理。

### 4.2 架构分层

建议将系统拆分为以下四层：

1. 展示层

- Web 门户前端
- 管理后台前端
- 知识库浏览与编辑界面

2. 业务层

- 用户与组织管理模块
- 知识库模块
- 权限管理模块
- 邮件同步与群组管理模块
- 审计与通知模块

3. 平台支撑层

- Keycloak 认证中心
- Casbin 权限引擎
- Redis 缓存与任务队列
- MinIO 附件存储
- Mailcow 邮件平台

4. 数据层

- PostgreSQL 主业务数据库
- MinIO 文件对象
- Mailcow 自身邮件存储

### 4.3 核心交互关系

- 用户通过前端访问系统，登录流程交由 Keycloak 统一处理。
- 业务后端负责管理用户资料、组织结构、知识库内容、文档权限、群组映射等业务数据。
- 权限判断由后端结合 Casbin 完成，支持角色、群组、方向、年级等多维授权。
- 文档编辑内容以 Markdown 为标准存储格式，编辑器采用 TipTap 进行增强输入和结构化编辑。
- 附件统一保存在 MinIO 中，数据库只保存元数据和访问策略。
- 群组与邮箱账号通过后端管理后台维护，再与 Mailcow 做同步。
- 异步任务如邮件通知、成员同步、搜索索引刷新等通过 Redis + BullMQ 处理。

## 5. 核心业务模块规划

### 5.1 用户与组织管理模块

功能范围：

- 用户注册
- 管理员审核
- 用户资料维护
- 方向分配
- 年级分配
- 角色分配
- 组织群组管理
- 账号状态控制

建议设计：

- 注册后账号初始状态为待审核。
- 审核通过后，管理员为成员指定方向、年级、角色和所属群组。
- 用户维度保留基础属性，如姓名、学号、邮箱、方向、年级、入学年份、身份状态等。

### 5.2 知识库模块

功能范围：

- 按方向创建知识库空间
- 空间下目录和页面管理
- Markdown 编辑与预览
- 版本历史
- 标签分类
- 页面引用
- 收藏与最近访问
- 审核与发布

建议设计：

- 顶层以空间为单位，例如 Android、iOS、Web、后端、鸿蒙、公共空间、年级空间。
- 空间下支持目录树和页面，满足结构化沉淀需求。
- 页面状态可分为草稿、待审核、已发布、已归档。
- 每篇文档保留创建人、修改人、发布时间、版本记录和权限信息。

### 5.3 权限管理模块

功能范围：

- 基于角色的权限控制
- 基于群组的权限控制
- 空间级权限
- 目录级权限
- 页面级权限
- 审核权限
- 管理后台权限

建议设计：

- 采用“角色 + 群组 + 资源”的混合授权模型。
- 默认以空间继承权限，必要时允许页面覆盖。
- 权限粒度建议包括查看、编辑、审核、发布、删除、管理五类核心动作。

### 5.4 邮箱与群组模块

功能范围：

- 实验室成员邮箱账号管理
- 邮件群组管理
- 群组抄送
- 方向组与年级组映射
- 系统邮件通知

建议设计：

- 群组邮箱包括方向维度和年级维度，例如 `android@lab.xxx`、`grade22@lab.xxx`。
- 群组关系以业务系统为主数据源，由后端同步到 Mailcow。
- 日报、周报、活动通知等场景可直接使用群组地址进行收件和抄送。

### 5.5 审计与通知模块

功能范围：

- 账号审核日志
- 权限变更日志
- 文档发布与删除日志
- 群组成员变更日志
- 系统邮件通知
- 站内消息通知

建议设计：

- 审计日志单独建模，确保关键管理操作可追溯。
- 邮件通知统一走异步任务，避免阻塞主业务请求。

## 6. 知识库内容模型与编辑方案

### 6.1 为什么采用 Markdown 作为核心存储格式

- 纯文本可读性强，便于长期保留与迁移。
- 适合技术文档、代码片段、配置说明、教程类内容。
- 可以方便导出、备份、版本管理。
- 与 GitHub、GitLab、开源社区写作习惯一致。

### 6.2 为什么需要扩展 Markdown 能力

标准 Markdown 对实验室知识库来说存在一些不足：

- 缺少结构化快捷录入体验。
- 不方便插入提示块、折叠块、模板块等富内容结构。
- 无法直接满足成员引用、页面引用、模板化日报周报等业务场景。

### 6.3 推荐的编辑方案

采用以下组合方案：

- 底层存储格式：Markdown
- 前端编辑器：TipTap
- 内容转换：TipTap JSON 与 Markdown 双向转换
- 扩展能力：自定义语法和结构块

### 6.4 建议支持的快捷写法

建议在标准 Markdown 基础上新增以下快捷能力：

- `/` 命令：快速插入标题、表格、代码块、提示块、折叠块、模板块。
- `@成员`：引用实验室成员。
- `@群组`：引用方向组、年级组或邮件群组。
- `[[页面名]]`：引用其他知识库页面。
- `:::note`、`:::warning`：提示块和警告块。
- 预设代码块模板：如 Java、Kotlin、Swift、Go、Python、前端组件模板。
- 日报和周报模板快捷插入。

### 6.5 内容存储建议

- Markdown 正文作为主存储格式。
- 结构化元数据单独存储在数据库中，如标题、标签、摘要、空间、目录、可见范围、审核状态等。
- 若需要更复杂的编辑器状态恢复，可额外保存 TipTap 的 JSON 快照作为编辑态缓存，但发布态以 Markdown 为准。

## 7. 认证与权限设计

### 7.1 认证中心采用 Keycloak

Keycloak 负责：

- 登录认证
- 账号生命周期管理
- 密码策略
- 邮箱验证
- Token 签发
- 单点登录扩展能力

采用 Keycloak 的原因：

- 避免自研认证安全细节。
- 支持标准协议，便于后续扩展。
- 社区成熟，适合 Docker 部署。

### 7.2 权限引擎采用 Casbin

Casbin 负责：

- 资源授权策略判断
- 角色、群组、资源之间的访问规则表达
- 复杂权限模型扩展

采用 Casbin 的原因：

- 能将权限规则从代码中抽离。
- 适合资源粒度复杂的知识库系统。
- 方便实现“空间继承 + 页面覆盖”的授权策略。

### 7.3 推荐权限模型

建议将权限拆为三层：

- 系统角色：超级管理员、方向管理员、年级管理员、普通成员。
- 组织群组：Android 组、iOS 组、Web 组、后端组、鸿蒙组、22 级、23 级等。
- 资源权限：针对知识库空间、目录、页面和管理功能的授权。

建议的基本授权规则：

- 普通成员可访问其有权限的知识库空间和页面。
- 方向管理员可管理所在方向空间与成员。
- 年级管理员可管理对应年级组织与通知场景。
- 超级管理员可管理全局账号、权限、系统配置和邮件同步。

## 8. 邮箱系统规划

### 8.1 为什么采用 Mailcow

Mailcow 是一套成熟的开源自建邮箱平台，适合组织内部邮箱使用，具备以下特点：

- 支持 Docker 化部署。
- 整合账号管理、Web 邮箱、SMTP、IMAP、反垃圾邮件等能力。
- 部署和维护难度低于自行拼装 Postfix、Dovecot、Webmail 方案。

### 8.2 邮箱系统在本项目中的定位

Mailcow 在本系统中主要承担以下职责：

- 管理成员邮箱账号
- 管理实验室邮件域名
- 管理邮件别名和群组邮箱
- 提供成员日常邮件收发能力

业务系统负责：

- 用户分组与组织结构维护
- 群组邮箱成员关系的主数据管理
- 审批后账号开通流程
- 与 Mailcow 的同步逻辑

### 8.3 群组邮箱规划

建议至少建立以下群组类别：

- 年级群组，如 `grade22@lab.xxx`、`grade23@lab.xxx`
- 方向群组，如 `android@lab.xxx`、`ios@lab.xxx`、`web@lab.xxx`
- 职能群组，如 `backend@lab.xxx`、`harmony@lab.xxx`
- 通知群组，如 `daily@lab.xxx`、`weekly@lab.xxx`

### 8.4 邮件同步策略

建议采用“业务系统主控，邮箱系统同步”的方式：

- 用户在业务系统注册并审核通过。
- 后端根据用户资料自动创建或更新邮箱账号。
- 群组成员变更时，由后端同步到对应邮件列表或别名配置。
- 所有同步任务通过异步队列执行，避免影响主流程。

## 10. Docker 部署规划

### 10.1 部署原则

- 所有核心服务容器化。
- 应用服务与邮件服务建议逻辑隔离。
- 配置、数据、日志统一挂载到宿主机持久化目录。
- 所有外部访问统一经过反向代理与 HTTPS。

### 10.2 推荐部署拆分

建议拆分为两套服务栈：

1. 应用业务栈

- Next.js 前端
- NestJS 后端
- PostgreSQL
- Redis
- MinIO
- Keycloak
- Traefik 或 Nginx Proxy Manager

2. 邮件服务栈

- Mailcow

### 10.3 为什么建议邮件服务独立部署

- 邮件系统对公网网络、域名解析、端口开放和 IP 信誉要求更高。
- 邮件服务维护与业务应用维护节奏不同。
- 独立部署可以减少相互影响，便于后续排查问题。

### 10.4 环境建议

起步阶段建议准备以下资源：

- 应用主机：4 核 8G 或以上
- 邮件主机：4 核 8G 或以上，建议独立公网 IP
- 存储空间：至少 200G SSD，并预留文档附件和邮件容量增长空间

### 10.5 域名与邮件基础要求

自建邮箱必须准备并正确配置以下基础能力：

- 域名解析
- SPF
- DKIM
- DMARC
- PTR 反向解析
- HTTPS 证书

若上述配置不完整，邮件可能被外部服务判定为垃圾邮件。

## 14. 风险与注意事项

### 14.1 邮件系统风险

- 自建邮箱不仅是代码问题，更依赖域名、DNS、IP 信誉和运维配置。
- 若服务器网络环境不佳，邮件到达率会明显受影响。

### 14.2 权限复杂度风险

- 知识库权限如果过度细化，会显著提高维护成本。
- 首版应以空间继承权限为主，页面级特例作为补充。

### 14.3 编辑器复杂度风险

- 不建议自行从零开发 Markdown 编辑器。
- 应在 TipTap 基础上做适度扩展，控制自定义范围。

### 14.4 运维复杂度风险

- 邮件服务与应用服务部署在一起会增加排障复杂度。
- 应尽量做好配置隔离、监控、备份和日志管理。

## 15. 最终推荐方案总结

本项目推荐采用以下总体方案：

- 前端：Next.js + TypeScript + Tailwind CSS + shadcn/ui
- 后端：NestJS + TypeScript + Prisma
- 主数据库：PostgreSQL
- 缓存与异步任务：Redis + BullMQ
- 附件存储：MinIO
- 认证中心：Keycloak
- 权限引擎：Casbin
- 知识库编辑器：TipTap + Markdown 转换链
- 邮箱系统：Mailcow
- 反向代理与 HTTPS：Traefik 或 Nginx Proxy Manager
- 部署方式：Docker Compose 起步

该方案兼顾了以下目标：

- 对实验室 100 人以内规模足够稳定
- 组件成熟，重复造轮子较少
- 适合分阶段建设
- 便于 Docker 部署和后续维护
- 既能满足知识沉淀，也能满足邮件协同需求
