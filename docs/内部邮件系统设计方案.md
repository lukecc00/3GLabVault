# 内部邮件系统设计方案

## 1. 目标

根据最新产品要求，实验室邮件能力调整为“仅限软件内部收发”的内部邮件系统，满足以下边界：

- 只能在本系统 Web / App 内查看、撰写、转发和归档邮件
- 不能接收任何外部邮件
- 不能向任何外部地址发送邮件
- 不允许被公网搜索引擎收录和搜索
- 不能暴露 SMTP / IMAP / POP3 等标准邮件协议入口
- 不依赖公网 MX 解析和外部邮件服务器

这意味着系统中的“邮箱地址”只作为实验室内部身份标识和站内信箱地址使用，不再代表一个可被互联网投递的真实邮箱。

## 2. 总体架构

内部邮件模块继续复用当前业务系统的主技术栈：

- 前端：Next.js 用户端 / 管理员端
- 后端：NestJS API
- 主库：PostgreSQL
- 缓存与异步任务：Redis
- 附件存储：MinIO

内部邮件的核心原则是“所有投递都在业务数据库和业务 API 内完成”，链路如下：

1. 发送人通过系统界面提交内部邮件
2. 后端校验发送人与收件人、群组、权限
3. 后端将群组收件人展开为具体用户
4. 后端在 PostgreSQL 中写入邮件主体、投递记录、审计记录
5. 收件人在本系统的“收件箱”中查看邮件

内部邮件主体投递链路不经过外部 SMTP 中继，也不需要 IMAP 拉取。

补充说明：

- 当前实现允许在内部邮件投递成功后，通过单独配置的外部 SMTP 账号向用户的真实外部邮箱发送提醒
- 该提醒邮件只用于提示“你有新的内部邮件待处理”，不包含正文、附件与详细收件信息
- 因此外部 SMTP 仅承担提醒职责，不参与内部邮件主链路

## 3. 安全边界

### 3.1 明确禁止的能力

为保证信息安全，生产环境应明确禁止以下能力：

- 不对公网搜索引擎开放索引
- 不配置公网 `MX`
- 不开放 `25 / 465 / 587 / 110 / 995 / 143 / 993` 等邮件协议端口
- 不部署 Mailcow、Postfix、Dovecot 等外部邮件收发组件作为正式方案
- 不允许用户填写外部收件地址并尝试投递
- 不提供“转发到个人邮箱”“邮件订阅到外部地址”“POP3/IMAP 客户端接入”这类能力

补充说明：

- 可以接入一个单独的 SMTP 发信账号，专门向用户的外部提醒邮箱发送纯提醒通知
- 该 SMTP 账号不能用于转发内部邮件正文，不能暴露为用户可配置的自由外发通道

### 3.2 系统内保留的能力

- 用户名和内部邮箱地址仍可作为登录标识
- 管理员可以按群组、年级、方向批量投递内部邮件
- 系统保留收件箱、发件箱、草稿箱、归档、回收站等内部信箱能力
- 邮件附件仅存储在 MinIO 私有桶，由鉴权接口控制访问
- 系统可在内部邮件送达后向用户外部提醒邮箱发送纯提醒通知

## 4. 地址规则

当前项目已经存在：

- `username`
- `email`

后续建议保留这两个字段，但重新定义其业务语义：

- `username`：登录账号
- `email`：内部邮件地址，例如 `zhangsan@3glab`

该地址的作用仅包括：

- 系统登录标识
- 内部邮件投递标识
- 成员列表展示

该地址不再具备：

- 对外收信能力
- 对外发信能力
- 与公网邮件客户端互通能力

## 5. 数据模型建议

建议在现有 Prisma 模型基础上新增以下内部邮件模型。

### 5.1 `InternalMailThread`

用于表示一封邮件会话或一条主题链。

建议字段：

- `id`
- `subject`
- `createdById`
- `lastMessageAt`
- `createdAt`
- `updatedAt`

### 5.2 `InternalMailMessage`

用于保存邮件正文和发送快照。

建议字段：

- `id`
- `threadId`
- `senderId`
- `subject`
- `bodyMarkdown`
- `bodyHtml`
- `sentAt`
- `isDraft`
- `replyToMessageId`
- `createdAt`
- `updatedAt`

### 5.3 `InternalMailRecipient`

用于保存每个收件人的独立投递状态。

建议字段：

- `id`
- `messageId`
- `userId`
- `recipientType`：`TO | CC | BCC`
- `deliverySourceType`：`USER | GROUP`
- `deliverySourceId`
- `readAt`
- `archivedAt`
- `deletedAt`
- `starredAt`

这样可以支持：

- 群组展开后的逐人已读状态
- 收件箱、抄送、密送分离
- 每个收件人的独立删除与归档

### 5.4 `InternalMailAttachment`

用于保存附件元数据。

建议字段：

- `id`
- `messageId`
- `fileName`
- `contentType`
- `size`
- `storageKey`
- `uploadedById`
- `createdAt`

附件实际内容继续放在 MinIO 私有存储。

### 5.5 `InternalMailAuditLog`

用于记录高风险操作。

建议字段：

- `id`
- `operatorId`
- `action`
- `targetType`
- `targetId`
- `detail`
- `createdAt`

建议至少审计：

- 群发邮件
- 删除草稿
- 彻底删除邮件
- 管理员代发或批量发送

## 6. 投递规则

### 6.1 允许的收件目标

只允许以下两类目标：

- 明确存在且状态为 `ACTIVE` 的内部用户
- 系统中已有的内部群组

### 6.2 群组投递

发送到群组时：

1. 后端根据群组展开成员列表
2. 自动去重
3. 过滤禁用、驳回或不存在的账号
4. 生成每个收件人的独立投递记录

### 6.3 拒绝外部地址

后端必须拒绝以下情况：

- 不存在于用户表的地址
- 域名不属于内部邮件域的地址
- 任何显式外部地址，如 `xxx@qq.com`
- 任何尝试通过 API 传入的自由文本外部地址

## 7. 权限模型建议

建议沿用当前角色和群组体系：

- 普通成员：可向任意内部成员发送，可抄送任意内部群组
- 方向管理员：可向任意内部成员发送，可抄送任意内部群组
- 年级管理员：可向任意内部成员发送，可抄送任意内部群组
- 超级管理员：可向任意内部成员发送，可抄送任意内部群组

后续如果需要更强治理，可以单独增加“群发审核”“发送频控”“审计预警”等管理策略，但不限制普通成员的跨组发送权限。

## 8. 前端界面规划

用户端建议新增：

- `收件箱`
- `发件箱`
- `草稿箱`
- `归档`
- `回收站`
- `写邮件`

管理员端建议新增：

- `内部邮件广播`
- `按群组投递`
- `按年级投递`
- `发送审计记录`

所有删除、清空、批量发送等危险操作，继续沿用现有的确认弹窗规则，必须输入指定确认文案后才能执行。

## 9. 与当前代码的衔接

当前仓库里已经存在一部分基于 Mailcow 的历史实现，包括：

- `MailcowService`
- `MAILCOW_*` 环境变量
- `mailboxProvisioningStatus` 等遗留字段

这些内容说明项目曾按“真实邮箱系统”方向推进过，但不应再作为最终上线方案。

迁移建议如下：

1. 第一阶段：文档与部署口径切换到内部邮件
2. 第二阶段：新增内部邮件 Prisma 模型与 API
3. 第三阶段：实现前端收件箱、发件箱、写信页
4. 第四阶段：停用 Mailcow 相关调用与环境变量
5. 第五阶段：清理遗留字段，完成数据迁移

在过渡期内，`User.email` 可以继续保留，作为内部邮件地址使用。

## 10. 上线要求

生产环境按内部邮件方案上线时，应至少满足：

- 站点默认 `noindex, nofollow`
- 仅允许登录后访问内部邮件页面和附件
- 业务域名仅暴露 Web/API 所需 HTTPS 入口
- 不暴露标准邮件协议端口
- 不配置公网邮件 DNS
- 附件桶为私有访问
- 邮件正文、附件下载、收件人列表全部走鉴权接口
- 管理员群发、删除、清空操作保留审计日志

如果启用外部提醒，还应额外满足：

- 外部提醒邮件不得包含内部邮件正文、附件和群组成员详情
- 提醒链路使用单独 SMTP 账号，不对用户开放自定义外发能力
- 提醒邮件正文只包含固定提示文案与系统入口链接
- 建议为同一用户增加提醒频控，避免短时间内重复轰炸

## 11. 下一步实施顺序

建议按以下顺序继续开发：

1. 冻结 Mailcow 方案，统一文档与环境变量说明
2. 设计 Prisma 新模型并生成迁移
3. 实现内部邮件模块的 DTO、Service、Controller
4. 先做“写信 + 收件箱 + 群组投递”最小闭环
5. 再补草稿、归档、附件、审计和搜索能力
