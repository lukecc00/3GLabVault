# 实验室知识库 + 邮箱系统开发计划书

## 1. 文档目的

本文档用于指导实验室知识库 + 邮箱系统项目的研发实施，明确项目目标、开发范围、阶段安排、任务拆解、人员分工、里程碑、验收标准和风险控制方案，为项目立项、过程推进和后续交付提供依据。

本文档基于已确定的技术路线：

- 前端：Next.js + TypeScript + Tailwind CSS + shadcn/ui
- 后端：NestJS + TypeScript + Prisma
- 主数据库：PostgreSQL
- 缓存与异步任务：Redis + BullMQ
- 附件存储：MinIO
- 认证中心：Keycloak
- 权限引擎：Casbin
- 知识库编辑器：TipTap + Markdown 转换链
- 邮箱系统：Mailcow
- 反向代理与 HTTPS：Traefik 或 Nginx Proxy Manager
- 部署方式：Docker Compose 起步

## 2. 项目背景

实验室当前需要一套内部统一平台，用于支撑以下典型场景：

- 按方向沉淀技术知识，包括 Android、iOS、Web、后端、鸿蒙等内容。
- 让实验室成员能够注册账号并参与内容建设。
- 支持按角色、年级、方向进行权限控制和组织管理。
- 提供实验室邮箱和群组邮箱，用于日报、周报、活动通知等邮件协作。
- 通过 Docker 降低部署与维护门槛，便于后续扩展和交接。

## 3. 项目目标

### 3.1 总体目标

建设一套适合 100 人以内实验室使用的统一协作平台，使其具备知识沉淀、成员管理、权限管控、邮件协作和基础运维能力。

### 3.2 业务目标

- 建立按方向划分的知识库空间。
- 建立统一的用户注册、审核、分组和权限管理流程。
- 支持 Markdown 为核心格式的知识录入，并提供快捷写法与模板能力。
- 建立实验室邮箱账号与群组邮箱体系。
- 支持方向组、年级组的邮件抄送与群发场景。

### 3.3 技术目标

- 基于成熟开源组件构建，减少重复造轮子。
- 采用模块化设计，便于后续迭代与交接维护。
- 采用 Docker Compose 进行起步部署。
- 保证基础安全性、可维护性和可扩展性。

## 4. 开发范围

### 4.1 本期纳入范围

- 项目基础工程与部署环境搭建
- 用户注册、登录、审核与状态管理
- 组织结构管理，包括方向、年级、群组、角色
- 知识库空间、目录、页面和版本管理
- Markdown 编辑器与快捷写法扩展
- 空间级和页面级权限控制
- 邮箱账号开通与群组邮箱映射
- 日报、周报模板和群组抄送支持
- 审计日志与基础通知能力

### 4.2 本期暂不纳入范围

- 实时协同编辑
- 复杂工作流引擎
- AI 自动问答和智能推荐
- 多终端 App
- 大规模微服务化拆分

## 5. 开发原则

- 先核心后增强：优先完成用户、权限、知识库和邮箱联动主流程。
- 先可用后完善：优先交付可运行、可部署、可演示的 MVP。
- 少造轮子：认证、邮箱、对象存储、反向代理优先使用成熟组件。
- 稳步推进：通过分阶段里程碑控制开发节奏和质量。
- 文档同步：每个阶段形成设计说明、接口说明、部署说明和测试结果。

## 6. 建议开发周期

建议采用 `12 周` 开发周期推进首个可用版本，分为 `6 个阶段`。

### 6.1 周期总览

- 第 1 周：项目启动与需求收敛
- 第 2 至 3 周：基础环境与底座搭建
- 第 4 至 5 周：用户、组织与权限模块开发
- 第 6 至 8 周：知识库模块开发
- 第 9 至 10 周：邮箱联动与通知能力开发
- 第 11 周：联调、测试、性能与安全检查
- 第 12 周：试运行、文档整理、验收与交付

说明：

- 如果人力较少，可将总周期调整为 14 至 16 周。
- 如果成员配置充足，可将部分前后端任务并行推进。

## 12. 风险分析与应对策略

### 12.1 邮件系统部署风险

风险说明：

- 邮件系统依赖公网 IP、域名、反向解析和反垃圾配置，复杂度高于普通 Web 服务。

应对策略：

- 邮件系统单独部署
- 提前完成域名和 DNS 方案准备
- 尽早验证 SPF、DKIM、DMARC、PTR 配置

### 12.2 权限复杂度风险

风险说明：

- 用户、角色、群组、空间、页面之间的授权规则容易膨胀。

应对策略：

- 首版以空间继承权限为主
- 页面级权限仅作特例补充
- 权限模型先少后多，避免过度设计

### 12.3 编辑器实现风险

风险说明：

- 自定义 Markdown 扩展和编辑器体验很容易超出预期工作量。

应对策略：

- 以 TipTap 为基础进行扩展
- 仅实现必要快捷写法
- 首版不做过深的自定义块体系

### 12.4 人力不足风险

风险说明：

- 实验室项目常见问题是开发者时间碎片化，容易拖慢进度。

应对策略：

- 控制首版范围
- 先交付 MVP
- 每周做阶段复盘和任务调整

### 12.5 需求变更风险

风险说明：

- 项目建设过程中容易新增“想顺手做”的功能。

应对策略：

- 明确版本边界
- 将新增需求放入下阶段待办
- 里程碑期间原则上不改动核心范围
