# 安全加固与开源组件方案

## 1. 目标

本项目面向实验室长期线上使用，安全目标分为三层：

- 保护网站与 API，降低暴力破解、常见 Web 攻击与错误配置风险
- 保护知识库内容，避免 Markdown / HTML 注入导致的 XSS
- 保护内部邮件系统，确保消息仅在系统内流转且不可被外部搜索

## 2. 当前已接入的开源安全组件

### 2.1 `helmet`

仓库：GitHub `helmetjs/helmet`

用途：

- 为 NestJS API 注入常见安全响应头
- 降低点击劫持、MIME 嗅探、部分浏览器侧攻击面

当前已在服务端 `main.ts` 接入。

### 2.2 `@nestjs/throttler`

仓库：GitHub `nestjs/throttler`

用途：

- 为登录、注册、改密、重置密码、批量生成账号等高风险接口提供限流
- 降低撞库、暴力破解、批量枚举风险

当前策略：

- 全局基础限流
- 登录接口更严格限流
- 注册、改密、管理员重置密码接口单独限流

### 2.3 `sanitize-html`

仓库：GitHub `apostrophecms/sanitize-html`

用途：

- 对 Markdown 渲染后的 HTML 做白名单清洗
- 去掉危险标签、危险属性与危险协议
- 降低知识库内容注入脚本的风险

当前已用于知识页渲染链路和编辑器初始内容转换链路。

## 3. 当前已接入的站点安全策略

### 3.1 Web 侧安全头

前端已通过 Next 配置增加：

- `X-Frame-Options: DENY`
- `X-Content-Type-Options: nosniff`
- `Referrer-Policy`
- `Permissions-Policy`
- `Cross-Origin-Opener-Policy`
- 基础 `Content-Security-Policy`

作用：

- 禁止站点被 iframe 嵌入
- 限制浏览器危险能力
- 为知识库正文渲染增加第二层浏览器策略保护

### 3.2 API 侧安全头

后端通过 `helmet` 输出常规 API 安全头，和 Web 端头策略形成前后端双层保护。

## 4. 知识库内容安全策略

知识库是当前最容易被忽视的高风险点，因为正文由用户输入，最终会变成 HTML。

当前策略如下：

1. 编辑器仍以 Markdown 为核心输入
2. 使用 `marked` 转换 Markdown
3. 使用 `sanitize-html` 清洗输出后的 HTML
4. 仅允许白名单标签、属性、链接协议
5. 外链统一补 `rel="noopener noreferrer nofollow"`

这意味着：

- 用户不能通过正文插入任意脚本
- `javascript:` 这类危险链接协议会被拦截
- 嵌入式攻击面显著降低

## 5. 内部邮件系统安全

### 5.1 当前边界

根据最新产品边界，邮件系统改为纯内部使用：

- 不线上开户
- 不接收外部邮件
- 不向外部发送邮件
- 不提供 SMTP / IMAP / POP3 访问
- 不允许被外部搜索引擎收录
- 仅允许内部用户互发和按群组抄送

因此安全重点不再是“对外邮件反垃圾”，而是“内部访问控制、内容隔离、附件保护和审计”。

### 5.2 推荐采用的安全策略

- 内部邮件页面必须登录后访问
- 附件存储在 MinIO 私有桶，不直接暴露公网链接
- 收件箱、发件箱、附件下载都走鉴权接口
- 群发、删除、清空等操作写入审计日志
- 前端页面增加 `noindex, nofollow`
- 如条件允许，部署在 VPN、校园网或内网网关后

说明：

- 内部邮件系统的主要攻击面从外部 SMTP 转为业务接口和附件访问
- 因此应继续复用现有 Web 安全组件，并在邮件模块增加更细粒度权限控制

## 6. 仍建议补充的开源安全组件

下面这些可以作为下一阶段增强项：

### 6.1 `nestjs-throttler-storage-redis`

用途：

- 将限流状态放到 Redis
- 当后端未来扩容为多实例时，仍然保持统一限流

适用场景：

- 生产多副本部署
- 多台机器或容器横向扩展

### 6.2 `Keycloak`

用途：

- 统一认证中心
- 支持更成熟的登录安全策略、账户管理与未来 SSO

适用场景：

- 实验室后续扩展到更多系统
- 希望统一账户、统一退出、统一 MFA

### 6.3 审计日志组件

可自建，也可接入开源日志栈，例如：

- `Grafana Loki`
- `Promtail`
- `Grafana`

用途：

- 记录管理员重置密码、批量建号、审核启停等高风险操作
- 在发生异常时可追溯

## 7. 安全上线建议

上线时建议至少满足以下要求：

- 管理后台只通过 HTTPS 暴露
- 数据库和 MinIO 不直接暴露公网端口
- 管理员初始化密码仅一次性使用
- 重要账号首次登录必须改密
- 定期轮换 `AUTH_TOKEN_SECRET`
- 定期审查管理员角色与群组权限
- 内部邮件页面设置为禁止搜索引擎索引
- 公网不开放标准邮件协议端口

## 8. 本项目当前推荐组合

如果目标是“100 人以内实验室、长期线上使用、尽量用成熟开源组件”，当前推荐组合是：

- Web / API 安全头：`helmet` + Next 安全 headers
- 接口防爆破：`@nestjs/throttler`
- 知识库内容净化：`sanitize-html`
- 内部邮件保护：鉴权接口 + MinIO 私有桶 + 审计日志
- 认证升级方向：`Keycloak`
- 审计与监控：`Grafana` + `Loki`

这套组合的优点是：

- 都是成熟开源项目
- 与当前栈兼容
- 可以逐步接入，不需要一次性大改架构
