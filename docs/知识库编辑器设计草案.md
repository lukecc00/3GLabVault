# 知识库编辑器设计草案

## 1. 目标

当前编辑器的目标是先满足知识库页面的可编辑、可保存和可展示，并为后续自定义 Markdown 快捷语法、模板块和权限控制预留扩展空间。

## 2. 当前技术方案

当前采用以下链路：

- 编辑器内核：`TipTap`
- 富文本基础扩展：`@tiptap/starter-kit`
- 编辑态转 Markdown：`turndown`
- 展示态 Markdown 渲染：`marked`

## 3. 当前工作流

### 3.1 编辑流程

- 页面进入编辑页后，从后端读取 `contentMd`
- 前端将 Markdown 转换为 HTML
- `TipTap` 使用 HTML 初始化编辑器内容
- 用户在编辑器中输入内容，支持快捷输入与工具栏操作

### 3.2 保存流程

- 编辑器更新时读取当前 HTML
- 前端使用 `turndown` 将 HTML 转换回 Markdown
- 同时读取 `editor.getJSON()` 得到结构化内容
- 保存时向后端提交：
  - `contentMd`
  - `contentRawJson`

### 3.3 展示流程

- 页面详情页从后端读取 `contentMd`
- 前端使用 `marked` 渲染为 HTML 展示

## 4. 当前支持能力

- 标题
- 段落
- 加粗
- 斜体
- 无序列表
- 有序列表
- 引用块
- 代码块

## 5. 当前快捷输入

依赖 `StarterKit` 当前可支持基础 Markdown 风格输入，例如：

- `#`：标题
- `-`：列表
- `>`：引用
- `````：代码块

## 6. 当前限制

当前版本仍是第一版编辑器，主要限制如下：

- 尚未实现自定义块，如 `note`、`warning`、`api`
- 尚未实现页面内目录树和锚点导航
- 尚未实现图片上传与附件插入
- 尚未实现多人协同编辑
- 尚未实现页面版本历史和差异对比
- 尚未实现 Markdown 扩展语法的双向精准转换

## 7. 后续扩展建议

下一步建议按下面顺序扩展：

- 增加页面更新审计与版本记录
- 增加自定义提示块和实验室模板块
- 增加附件上传与图片插入
- 增加 `[[页面引用]]`、`@成员`、`@群组`
- 增加目录树、页面层级与拖拽排序
- 增加页面历史版本和回滚

## 8. 结论

当前方案的特点是：

- 输入体验优于纯 Markdown 文本框
- 数据存储仍以 Markdown 为主，便于迁移和导出
- 结构化 JSON 作为增强字段，为后续编辑器升级留出空间

这套方案适合作为实验室知识库第一版编辑器基础设施。
