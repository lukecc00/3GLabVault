# 实验室知识库 + 邮箱系统数据库模型设计草案

## 1. 文档目的

本文档用于说明当前首版后端数据模型设计，重点覆盖用户、角色、群组和组织管理相关实体，为后续注册审核、权限控制、知识库与邮箱联动开发提供基础。

## 2. 当前建模范围

当前已在 `Prisma` 中完成以下实体建模：

- `User`：用户
- `Role`：角色
- `UserRole`：用户与角色关联
- `Group`：群组
- `UserGroupMembership`：用户与群组关联
- `KnowledgeSpace`：知识库空间
- `KnowledgePage`：知识库页面

## 3. 枚举设计

### 3.1 用户状态

`UserStatus`

- `PENDING`：待审核
- `ACTIVE`：已启用
- `DISABLED`：已禁用
- `REJECTED`：已驳回

### 3.2 群组类型

`GroupType`

- `DIRECTION`：方向组，如 Android、iOS、Web、后端、鸿蒙
- `GRADE`：年级组，如 22 级、23 级
- `FUNCTIONAL`：功能组，如审核组、知识库编辑组
- `SYSTEM`：系统内部群组

### 3.3 群组成员身份

`MembershipRole`

- `MEMBER`：普通成员
- `MANAGER`：群组管理员

### 3.4 空间可见性

`SpaceVisibility`

- `PUBLIC`：公开空间
- `PRIVATE`：私有空间
- `GROUP_RESTRICTED`：仅特定群组可见

### 3.5 页面状态

`PageStatus`

- `DRAFT`：草稿
- `PUBLISHED`：已发布
- `ARCHIVED`：已归档

## 4. 实体说明

### 4.1 User

用于表示实验室成员账号主数据。

核心字段：

- `id`：主键
- `username`：用户名，可空，后续可用于平台昵称或登录标识
- `email`：邮箱，唯一
- `realName`：真实姓名
- `studentId`：学号，唯一，可空
- `keycloakUserId`：认证中心用户 ID，唯一，可空
- `status`：用户状态
- `avatarUrl`：头像地址
- `bio`：个人简介
- `createdAt` / `updatedAt`：创建与更新时间

关系：

- 一个用户可拥有多个角色
- 一个用户可加入多个群组

### 4.2 Role

用于表示系统角色。

核心字段：

- `id`：主键
- `code`：角色编码，唯一
- `name`：角色名称
- `description`：角色描述
- `isSystem`：是否系统预置角色

建议首批预置角色：

- `SUPER_ADMIN`
- `DIRECTION_ADMIN`
- `GRADE_ADMIN`
- `MEMBER`

### 4.3 UserRole

用于建立用户与角色的多对多关系。

核心字段：

- `userId`
- `roleId`
- `assignedAt`

约束：

- 同一用户与同一角色不可重复绑定

### 4.4 Group

用于表示组织群组。

核心字段：

- `id`：主键
- `code`：群组编码，唯一
- `name`：群组名称
- `type`：群组类型
- `description`：描述
- `parentId`：父群组 ID，可空

说明：

- 该表同时承载方向组、年级组、功能组等组织概念
- 支持父子层级，便于后续扩展院系、实验室、子方向等结构

### 4.5 UserGroupMembership

用于建立用户与群组的多对多关系。

核心字段：

- `userId`
- `groupId`
- `membershipRole`
- `joinedAt`

约束：

- 同一用户与同一群组不可重复绑定

### 4.6 KnowledgeSpace

用于表示知识库顶层空间，例如 `Android 空间`、`公共空间`。

核心字段：

- `code`：空间编码，唯一
- `slug`：空间标识，唯一
- `name`：空间名称
- `description`：空间描述
- `visibility`：空间可见性
- `ownerGroupId`：归属群组，可空

说明：

- 一个空间可绑定到一个方向组或年级组
- 一个空间下可创建多篇知识页面

### 4.7 KnowledgePage

用于表示具体知识库页面。

核心字段：

- `spaceId`：所属空间
- `authorId`：作者，可空
- `editorId`：最后编辑者，可空
- `title`：页面标题
- `slug`：页面标识，在空间内唯一
- `summary`：页面摘要
- `contentMd`：Markdown 正文
- `contentRawJson`：后续可承载编辑器结构化数据
- `tags`：标签数组
- `status`：页面状态
- `publishedAt`：发布时间，可空

说明：

- 当前存储以 `Markdown` 为主
- 后续可在不破坏兼容性的前提下叠加 `TipTap` 结构化内容

## 5. 关系说明

当前模型关系如下：

- `User 1 - n UserRole`
- `Role 1 - n UserRole`
- `User 1 - n UserGroupMembership`
- `Group 1 - n UserGroupMembership`
- `Group 1 - n Group`，用于父子群组层级
- `Group 1 - n KnowledgeSpace`
- `KnowledgeSpace 1 - n KnowledgePage`
- `User 1 - n KnowledgePage`，分别作为作者和编辑者

## 6. 当前接口骨架

当前后端已提供以下基础查询接口骨架：

- `GET /api/users`
- `GET /api/roles`
- `GET /api/groups`
- `GET /api/organizations/summary`
- `GET /api/knowledge/spaces`
- `GET /api/knowledge/spaces/:id`
- `POST /api/knowledge/spaces`
- `GET /api/knowledge/pages`
- `GET /api/knowledge/pages/:id`
- `POST /api/knowledge/pages`

这些接口当前已经可以支撑知识库空间和页面的首版创建与查询，后续会继续扩展更新、审核、权限控制、版本历史和目录结构。

## 7. 后续扩展建议

下一步建议基于当前模型继续扩展：

- 用户注册申请表或审核记录表
- 权限策略表或 Casbin 策略落库方案
- 知识目录节点表与页面树结构
- 知识页面版本历史表
- 邮箱账号映射表
- 邮箱群组同步记录表
- 审计日志表

## 8. 结论

当前模型已经能够承载“用户 - 角色 - 群组 - 知识空间 - 知识页面”这条主线，是整个实验室系统后续实现账号审核、组织管理、权限分配、知识沉淀和邮箱群组映射的基础数据层。
